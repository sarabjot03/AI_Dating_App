# Phase 0 — AI profile generation (product rules)

**Status:** Approved for v1 implementation  
**Last updated:** 2026-05-16  
**Owners:** Product + engineering (Spark / AI Dating App)

This document is the contract for Phases 1–5. Code constants live in:

- `services/api/src/profile-generation/constants.ts`
- `apps/mobile/constants/profile-generation.ts`

---

## 1. User journey (order is fixed)

| Step | Screen | User action | System |
|------|--------|-------------|--------|
| 1 | Questionnaire | Answer all questions + name, city, one-liner | Save answers locally + optional draft sync |
| 2 | Photo upload | Select **1–10** photos | Upload to object storage; show progress |
| 3 | AI building | Tap **Generate my profile** (when min photos met) | Server runs moderation → pick up to 5 → prompts → captions → bio |
| 4 | Preview | Review & edit everything | All fields are **drafts** |
| 5 | Go live | Tap **Save & enter Spark** | Persist profile, set `onboardedAt`, enter app |

**Rule:** User cannot skip to Discover until step 5 succeeds.

---

## 2. Questionnaire (step 1)

- Uses existing compatibility questionnaire (`version` from schema JSON).
- **Required before photo upload:** all questionnaire steps + `displayName` (≥2 chars), `city` (≥2), `aboutLine` (≥8 chars).
- Questionnaire answers are the **source of truth** for AI text; photos are the source of truth for visual ranking and captions.

---

## 3. Photos (step 2)

### 3.1 Counts

| Constant | Value | Meaning |
|----------|-------|---------|
| `PHOTOS_UPLOAD_MIN` | **3** | Minimum uploads before AI can run (quality bar for dating) |
| `PHOTOS_UPLOAD_MAX` | **10** | Maximum uploads per user |
| `PHOTOS_AI_SELECTED_MAX` | **5** | AI picks **up to 5** (if user uploaded 4, all 4 may be selected) |
| `PHOTOS_FEED_PRIMARY_COUNT` | **1** | Discover card hero = first selected photo |

**Confirmed (product):** User may upload **any number from 3 to 10** (not required to fill all 10 slots).

### 3.2 “Any size” (clarified for engineering)

- **User-facing:** “Upload your best 10 photos—we’ll handle the rest.”
- **Technical:**
  - **Original** may be up to **25 MB** per file at storage (server limit).
  - **Client** resizes before upload (max width **1600 px**, JPEG quality **~0.85**) to keep uploads fast.
  - **AI** receives **1024 px** max edge thumbnails (URLs), not full originals.

### 3.3 Allowed formats

- JPEG, PNG, HEIC (HEIC converted to JPEG on server in Phase 1).
- Reject GIF, WebP (v1), PDF, video.

### 3.4 Content rules (user-visible)

- Must be **of the user** (face visible in most shots recommended; AI will deprioritize no-face shots).
- **No** nudity, lingerie-only shots, or sexually explicit content.
- **No** weapons, hate symbols, illegal activity, or minors.
- **No** heavy filters that obscure face; **no** watermarked stock photos.
- **Group photos** allowed but AI should not pick more than **2** group shots in the final 5.

---

## 4. Moderation (before AI runs)

**When:** After user has ≥ `PHOTOS_UPLOAD_MIN` uploads and taps generate, **before** photo ranking or text generation.

**v1 approach:**

1. Automated check on each uploaded image (provider safety API or vision model safety).
2. If **any** image fails → block generation; show which slot failed + “Replace this photo”.
3. Log moderation outcome server-side (no public detail).

**Hard fail:** sexual content, violence, CSAM risk, hate.  
**Soft fail (warn):** no detectable face → still allow upload but lower score in ranking.

---

## 5. AI outputs (steps 3–4)

All outputs are **drafts** until the user taps **Save & enter Spark**.

### 5.1 Photo selection (N → up to 5)

- AI ranks all uploaded photos (N = 3–10) on: face clarity, lighting, solo vs group, variety, authenticity.
- Output: ordered list of **`min(N, 5)`** `photoId`s + optional short internal reasons (not shown to user in v1).

### 5.2 Hinge-style prompts (from questionnaire only)

| Constant | Value |
|----------|-------|
| `PROMPT_COUNT` | **3** |

- Each prompt: `{ question: string, answer: string }`.
- Questions generated by AI (Hinge-style), answers grounded in questionnaire (no inventing facts).
- Length: question 10–80 chars, answer 20–200 chars.

**Example templates (AI may vary wording):**

- “A green flag I look for…”
- “My perfect Sunday…”
- “We’ll get along if…”

### 5.3 Photo captions (from selected 5 + questionnaire tone)

| Constant | Value |
|----------|-------|
| `CAPTION_MAX_LENGTH` | **80** characters |

- One caption per selected photo.
- Tone: **flirty, catchy, warm** — never explicit, never crude.
- Must relate to what’s visible in the photo; may lightly reference personality from questionnaire.

### 5.4 Bio (from questionnaire only)

| Constant | Value |
|----------|-------|
| `BIO_MIN_LENGTH` | **40** |
| `BIO_MAX_LENGTH` | **280** |

- **2–3 short sentences**, first-person.
- Tone: flirty, confident, specific (city, vibe, intent) — grounded in answers.
- No lists of demands; no negativity about exes.

---

## 6. Tone & safety (all text)

**Do:**

- Playful, confident, specific, inclusive.
- Light humor; dating-app appropriate.

**Don’t:**

- Explicit sexual content, slurs, harassment, medical/financial claims.
- “AI generated” disclaimer on profile (optional in settings only).

**Regenerate:** User may tap **Regenerate profile** on preview screen, max **2** times per 24 hours (rate limit on API).

---

## 7. User control (preview screen)

User **can edit** before go-live:

- All **5** photo order (drag to reorder among the selected 5 only in v1).
- All **3** prompt answers (questions fixed or editable — v1: **answers editable**, questions editable).
- All **5** captions.
- Bio.
- Display name (from questionnaire, editable here).

User **cannot** in v1:

- Swap in a non-selected photo from the upload set without re-running AI (Phase 2 feature: “Pick different photos”).
- Publish with zero selected photos (must keep at least 1 after AI run).

**Save** validates lengths and required fields; then same as today’s `PATCH /v1/me/profile` + new photo records.

---

## 8. LLM & storage (v1 — confirmed)

| Role | Choice |
|------|--------|
| Object storage | **Cloudinary** |
| Language | **English only** (bios, prompts, captions) |
| LLM | **Google Gemini only** (`gemini-2.0-flash` or current Flash multimodal) |
| Fallback provider | **None** in v1 |

**Why Gemini for v1:** one API for ranking + captions + bio + prompts; strong vision; cost-effective on Render.

**Secrets:** `GEMINI_API_KEY` and Cloudinary credentials only on server (Render env). Never in Expo bundle.

---

## 9. Non-goals for Phase 0 / v1

- Embeddings / vector search for matching.
- Video or voice prompts.
- AI chat on profile.
- Automatic publish without user Save.
- Storing full-resolution images in Postgres (use Cloudinary).

---

## 10. Success metrics (Phase 0 baseline)

- ≥ **80%** of users who upload ≥3 photos complete Save within one session.
- Moderation false-positive rate tracked; target &lt; **5%** manual review later.
- Median generation time &lt; **60 s** (p95 &lt; 120 s).
- User edit rate on bio (how many change AI bio before Save) — informational.

---

## 11. Confirmed product decisions (2026-05-16)

| # | Decision |
|---|----------|
| 1 | **3–10 photos** per user (max 10); AI selects up to 5 |
| 2 | **Cloudinary** for media storage |
| 3 | **English only** for generated copy in v1 |
| 4 | **Gemini only** for LLM in v1 (no OpenAI fallback yet) |
